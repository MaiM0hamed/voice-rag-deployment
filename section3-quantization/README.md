A benchmark harness comparing **`Qwen/Qwen2.5-1.5B-Instruct`** at full precision
(FP16/BF16) against **4-bit NF4 quantization** (bitsandbytes, double quant),
measuring memory, load time, TTFT, throughput, total latency and output quality
across five fixed prompts. Both variants use the **same model, tokenizer,
generation settings and prompts** — only weight precision differs.

**Every number is measured at runtime by `src/benchmark.py`. Nothing is
estimated, interpolated, or hard-coded.** Results are produced by running the
benchmark on your hardware (a free Colab T4 runs both variants); they are not
committed pre-filled.

## Installation

```bash
cd section3-quantization

# 1. Install torch for YOUR hardware first
pip install torch --index-url https://download.pytorch.org/whl/cu121   # GPU (CUDA 12.1)
# pip install torch --index-url https://download.pytorch.org/whl/cpu   # CPU only

# 2. Install the rest
pip install -r requirements.txt
```

First benchmark run downloads `Qwen2.5-1.5B-Instruct` (~3 GB) into
`~/.cache/huggingface`; later runs are offline.

## Environment variables

Copy `.env.example` to `.env` to override any default (all optional):
`MODEL_ID`, `MAX_NEW_TOKENS` (128), `DO_SAMPLE` (false), `TEMPERATURE` (0.0),
`REPEATS` (3), `WARMUP_TOKENS` (16).

## Run the benchmark

### Google Colab (recommended — runs BOTH variants on a free T4)

Click the **Open in Colab** badge above, set *Runtime → Change runtime type →
T4 GPU*, and run all cells. It clones the repo, installs deps, benchmarks FP16
then INT4, writes both artifacts, displays the summary, and offers them for
download. See `colab_benchmark.ipynb`.

### Local

```bash
python results/generate_results.py                 # both variants (needs a CUDA GPU for INT4)
python results/generate_results.py --skip-int4     # FP16 only (works on CPU)
python results/generate_results.py --skip-fp16     # INT4 only
```

On a CPU-only machine, FP16 runs (in float32) and the 4-bit variant is **skipped
with the reason recorded** in the report — never filled in with a guess. For CPU
quantization use the GGUF/llama.cpp route described in `NOTES.md`.

## Generated artifacts

| File | Produced by | Committed? | Contents |
|---|---|---|---|
| `results/comparison_report.md` | `generate_results.py` / Colab notebook | Yes, after a real run | Environment, methodology, comparison table, side-by-side FP16-vs-INT4 outputs for all 5 prompts |
| `results/raw_measurements.json` | same | Yes, after a real run | Per-prompt memory, timings, token counts and generated text for both variants |

Both are `.gitignore`-permitted (see `.gitignore`) and should be committed once
generated on the target hardware. Do not commit a run from mismatched hardware.

## Testing

```bash
python -m pytest tests/ -q     # 6 offline tests; torch/transformers stubbed, no downloads
```

These verify the report arithmetic, ratio formatting, missing-variant handling
and divide-by-zero guards. They do **not** produce benchmark numbers — real
measurements come only from running the benchmark on hardware.

## Limitations

- 4-bit NF4 requires CUDA (bitsandbytes kernels are GPU-only). CPU machines get
  the FP16 baseline plus a recorded skip for INT4.
- Single model at a single size (1.5B); quantization damage grows as models
  shrink, so results should not be extrapolated upward without re-measuring.
- Qualitative quality is judged by reading the five side-by-side outputs; no
  automated quality score is invented over such a small sample.
