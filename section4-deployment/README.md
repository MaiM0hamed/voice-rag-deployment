# Section 4 — Model Deployment (FastAPI + Docker)

A containerised FastAPI service that serves `Qwen/Qwen2.5-1.5B-Instruct` behind a
REST API with a health probe, a blocking completion endpoint, and a genuinely
token-by-token streaming endpoint (Server-Sent Events). It includes structured
logging, configuration via environment variables, error handling, a test suite,
and load-testing tooling that measures TTFT and latency under concurrency.

The detailed engineering write-up — why FastAPI over vLLM/TGI, the streaming
design, and the 50-concurrent-user scaling plan — is in
[`NOTES.md`](NOTES.md). This README is the operational guide.

## Architecture

```
Client ──HTTP──▶ FastAPI (app/main.py)
                 ├─ GET  /health   readiness (ok / loading / error)
                 ├─ POST /chat     blocking completion (JSON)
                 └─ POST /stream   SSE, token-by-token
                        │
                        ▼
               InferenceEngine (app/inference.py)
                 ├─ model loaded ONCE at startup (FastAPI lifespan)
                 ├─ generate()  → asyncio.to_thread (off the event loop)
                 ├─ stream()    → Thread + TextIteratorStreamer → run_in_executor
                 └─ asyncio.Semaphore(MAX_CONCURRENT_REQUESTS) bounds concurrency
                        │
                        ▼
               transformers (Qwen2.5-1.5B-Instruct), optional 4-bit NF4 on GPU
```

`transformers.generate()` is blocking, so it runs off the event loop; otherwise
one request would stall health checks and every other request. Streaming uses
`TextIteratorStreamer` for a real low time-to-first-token, not a full response
buffered and chunked afterwards.

## Installation

```bash
cd section4-deployment

# Install torch for your hardware FIRST (kept out of requirements.txt on purpose)
pip install torch --index-url https://download.pytorch.org/whl/cpu    # CPU
# pip install torch --index-url https://download.pytorch.org/whl/cu121  # GPU

pip install -r requirements.txt
```

First startup downloads `Qwen2.5-1.5B-Instruct` (~3 GB) into the HuggingFace
cache; later starts are fast.

## Environment variables

Copy `.env.example` to `.env` to override (all optional):

| Variable | Default | Purpose |
|---|---|---|
| `MODEL_ID` | `Qwen/Qwen2.5-1.5B-Instruct` | HuggingFace model to serve |
| `LOAD_IN_4BIT` | `false` | Serve the Section 3 4-bit NF4 build (requires CUDA) |
| `MAX_NEW_TOKENS` | `256` | Default generation cap (per-request override allowed) |
| `DEFAULT_TEMPERATURE` | `0.7` | Default sampling temperature (0 = greedy) |
| `HOST` / `PORT` | `0.0.0.0` / `8000` | Bind address |
| `LOG_LEVEL` | `INFO` | Log verbosity |
| `MAX_CONCURRENT_REQUESTS` | `4` | Bounds in-flight generations (semaphore) |
| `REQUEST_TIMEOUT_S` | `120` | Per-request timeout |

## Running with Docker

```bash
docker build -t electropi-inference .
docker run -p 8000:8000 electropi-inference

# GPU (needs the NVIDIA Container Toolkit), serving the 4-bit build:
docker build --build-arg TORCH_INDEX=https://download.pytorch.org/whl/cu121 -t electropi-inference:gpu .
docker run --gpus all -e LOAD_IN_4BIT=true -p 8000:8000 electropi-inference:gpu
```

## Running with Docker Compose

```bash
docker compose up --build            # CPU
docker compose --profile gpu up      # GPU (NVIDIA Container Toolkit)
```

The compose file persists the HuggingFace cache in a named volume so weights are
downloaded once across rebuilds.

## Endpoints

### Health endpoint

`GET /health` → `200` always (so an orchestrator can read readiness):

```json
{"status": "ok", "model": "Qwen/Qwen2.5-1.5B-Instruct", "model_loaded": true,
 "device": "cpu", "quantized": false, "uptime_s": 12.3}
```

`status` is `loading` while weights are not yet resident, `ok` once ready — this
lets a load balancer distinguish "starting" from "broken".

### Generate endpoint

`POST /chat` → complete response with token usage and server-side latency.
Returns `503` if the model is still loading, `422` on invalid input.

### Streaming endpoint

`POST /stream` → `text/event-stream`. Each event is `data: {"token": "..."}`;
the stream terminates with `data: [DONE]`. Errors after streaming has begun are
delivered in-band as `data: {"error": "..."}` (the HTTP status is already
committed once the first byte is sent).

## Example curl requests

```bash
# Health
curl localhost:8000/health

# Blocking completion
curl -X POST localhost:8000/chat \
  -H 'Content-Type: application/json' \
  -d '{"messages":[{"role":"user","content":"What is a Raspberry Pi?"}],"max_new_tokens":128}'

# Streaming (SSE)
curl -N -X POST localhost:8000/stream \
  -H 'Content-Type: application/json' \
  -d '{"messages":[{"role":"user","content":"Count to five."}]}'
```

## Example Python client

```python
import httpx, json

BASE = "http://localhost:8000"

# Blocking
resp = httpx.post(f"{BASE}/chat", json={
    "messages": [{"role": "user", "content": "What is an ADC?"}],
    "max_new_tokens": 128,
}, timeout=120)
print(resp.json()["content"])

# Streaming
with httpx.stream("POST", f"{BASE}/stream", json={
    "messages": [{"role": "user", "content": "Count to five."}],
}, timeout=120) as r:
    for line in r.iter_lines():
        if line.startswith("data: "):
            body = line.removeprefix("data: ")
            if body == "[DONE]":
                break
            event = json.loads(body)
            print(event.get("token", event.get("error", "")), end="", flush=True)
```

## Load testing

Start the service first (locally or via Docker), then:

```bash
# Async burst: N simultaneous streaming requests (the assessment requires 10)
python results/generate_results.py                  # 10 concurrent users
python results/generate_results.py --users 1 10 25  # concurrency sweep

# or the raw script
python loadtest/benchmark.py --users 10 --url http://localhost:8000

# Sustained traffic with Locust (live UI or headless)
locust -f loadtest/locustfile.py --headless -u 10 -r 10 -t 60s \
       --host http://localhost:8000 --html results/locust_report.html
```

This writes `results/load_test_report.md` (TTFT/latency/throughput tables) and
`results/load_test_results.json` (per-request timings). TTFT is measured against
`/stream` (the only honest way — on `/chat` the first byte arrives only when
generation completes). See [`results/README.md`](results/README.md).

## Testing

```bash
python -m pytest tests/ -q     # 9 tests, mocked engine, no weights/downloads
```

Covers routing, JSON validation (`422`), `503` while unloaded, SSE framing +
`[DONE]`, mid-stream error delivery, and the `X-Accel-Buffering` header.

## Troubleshooting

| Symptom | Cause / fix |
|---|---|
| `/health` shows `"model_loaded": false` | Weights still downloading (first run ~3 GB). Wait; watch logs. |
| `503` from `/chat` or `/stream` | Model not loaded yet — poll `/health` until `ok`. |
| `LOAD_IN_4BIT=true` errors on load | 4-bit needs a CUDA GPU (bitsandbytes is GPU-only). Set `false` for CPU. |
| Container killed during startup | First download can exceed a short healthcheck window; `start-period` is 300 s by design — give it time / more RAM. |
| Streaming buffers instead of trickling | A proxy is buffering; the service already sends `X-Accel-Buffering: no` — configure nginx/ingress accordingly. |
| OOM on CPU | Full-precision 1.5B needs ~6 GB RAM. Use a machine with more RAM or a GPU with `LOAD_IN_4BIT=true`. |

## Reproducibility

All load-test numbers are produced by `loadtest/benchmark.py` against a live
service and written to `results/`; none are estimated or hand-entered. The test
suite is deterministic and hardware-independent (mocked engine). Pin versions
via `requirements.txt`; the Docker image fixes the runtime.

## Limitations

- Single node: no gateway, request queue, or replica set — the scaling design
  for 50 concurrent users is written up in `NOTES.md` but deliberately not built.
- No authentication or rate limiting (out of scope for this task; called out
  rather than left implied).
- HuggingFace `generate()` has no continuous batching; throughput saturates well
  before 50 users. `NOTES.md` details the vLLM migration path.
- 4-bit serving requires a CUDA GPU.
